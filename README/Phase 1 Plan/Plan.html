1. Objective

Build a portfolio-grade full-stack application that demonstrates the ability to design relational data models, implement end-to-end CRUD workflows, structure a scalable backend using Next.js + Prisma + PostgreSQL, model a real-world clinical research workflow, and deliver a complete, demoable system.

Primary goal: maximize hireability by showcasing practical engineering competence, disciplined scope control, and product thinking.
Optimization rule: adopt high-signal, low-cost improvements that materially raise interviewer confidence, and reject anything that delays finishing the core system.

2. Scope Definition

This project is not a full CTMS. It is a focused Clinical Research Study Tracker built around the core operational workflow that is most useful for demonstrating engineering skill.

Included

  • Studies
  • Sites
  • Participants
  • Visit scheduling and tracking
  • Document management
  • Task management
  • Dashboard metrics
  • Seed data for demo reliability
  • Basic validation and status enums

Explicitly Excluded

  • Full IRB/regulatory lifecycle workflows
  • Adverse event reporting
  • Dynamic eCRF / CRF builder
  • 21 CFR Part 11 e-signature workflow
  • Billing and finance modules
  • Complex permission matrices
  • Advanced audit/compliance engines
  • Anything that expands scope before the vertical slice is finished
Rationale: reducing scope is not a weakness here. It is evidence of judgment. A smaller, finished product creates a better hiring signal than a much larger incomplete platform.

3. System Architecture

Frontend

Next.js App Router with React components and Tailwind styling.

Backend

Next.js route handlers for APIs and domain-specific CRUD workflows.

Data Layer

Prisma ORM over PostgreSQL with a single readable schema.

Architecture Pattern

UI (React / Next.js) ↓ API Routes (Next.js) ↓ Zod Validation ↓ Prisma Client ↓ PostgreSQL

Key Design Principles

  • Single Prisma schema for simplicity and maintainability
  • Relational-first design with clean foreign keys and normalized structure
  • CRUD-first development before advanced features
  • Vertical slice completion over feature sprawl
  • Readable over clever
  • Store dates in UTC and convert for display at the UI layer
  • Use transactions for multi-step writes that must succeed atomically

4. Core Data Model

The MVP data model should be strong enough to represent a believable clinical research workflow while remaining small enough to finish and explain clearly.

User / Role

Supports simple ownership, seeded demo users, and lightweight authorization checks.

Study / Site

Captures protocol-level organization and site-level operational structure.

Participant

Tracks enrollment state and links to study and site context.

VisitTemplate

Defines expected study visits such as screening or follow-up windows.

ParticipantVisit

Represents actual visit instances and completion status.

Document / Task

Models real operational work: file management and execution tracking.

Highest-Value Modeling Decision

Template vs. Instance separation: VisitTemplate stores protocol expectation, while ParticipantVisit stores actual execution. This avoids duplication, supports scalable scheduling logic, and demonstrates the difference between business rules and real-world state.

Relationship Model

  • Study → many Sites
  • Study → many Participants
  • Site → many Participants
  • Participant → many ParticipantVisits
  • Study → many VisitTemplates
  • VisitTemplate → many ParticipantVisits
  • Study / Participant → many Documents
  • User → many Tasks

State Integrity Rules

  • Use enums for participant, visit, and task statuses instead of free-form strings
  • Keep document modeling simple: metadata in PostgreSQL, file stored externally
  • Do not add document versioning until the core workflow is complete

5. Development Strategy

The right build strategy is not “build everything.” It is staged execution with a clear stopping rule.

Phase 1 — Foundation (Proof of Viability)

  • Set up Next.js, Prisma, and PostgreSQL
  • Define initial schema
  • Run migrations
  • Confirm basic CRUD through route handlers
  • Add Zod request validation at API boundaries
  • Use enums for stateful fields from day one

Deliverable: working backend with core models wired correctly.

Phase 2 — Vertical Slice (Core Workflow)

1
Create Study
Establish the top-level protocol container.
2
Add Site
Create site-level operational context.
3
Enroll Participant
Track subject status within study and site.
4
Assign Visit Templates
Define expected protocol visits.
5
Generate Participant Visits
Create actual visit records and statuses using a single transaction.
6
Upload Document
Attach operational content to study or participant.
7
Create Task
Track execution work and ownership.

Deliverable: one full workflow that can be demonstrated end to end.

Phase 3 — Productization (Minimum Marketable Version)

  • Dashboard with key metrics
  • Filtering and search for studies, participants, and tasks
  • Status badges and better UX states
  • Seeded demo data for reliable presentation
  • Optional: basic auth with minimal role checks, only if it does not delay completion
  • Optional: lightweight audit log for a few high-value actions, only after the core workflow is done

Deliverable: a polished application that feels real, not purely technical.

6. UI Structure

Demo Path Requirement

8. Interview Talking Points

Why this project?

I wanted a project aligned with my clinical research background, but scoped to show strong full-stack fundamentals instead of trying to build a full CTMS and getting buried in complexity.

Why this architecture?

Next.js with Prisma and PostgreSQL gives strong type safety, clean data access, and fast iteration, which makes it a good fit for a portfolio project that needs both technical credibility and execution speed.

Why not include adverse events, IRB workflows, or a dynamic form builder?

Those are platform-level complexity multipliers. I intentionally focused on workflow integrity, relational modeling, and end-to-end execution because those provide a stronger hiring signal per unit effort.

What higher-signal engineering details are included?

  • Visit generation is done transactionally to prevent partial enrollment state
  • Status fields use enums to prevent data drift
  • Dates are stored in UTC and converted for display
  • Zod validates request bodies before they reach the database layer

How would this extend later?

  • Add adverse event tracking
  • Introduce role-based permissions
  • Support document versioning and audit logging
  • Expand toward dynamic CRF workflows

9. Risk Management

Risk Why It Matters Mitigation
Over-scoping The project becomes too large to finish cleanly. Hold scope at the study tracker level.
Schema creep Too many tables and edge cases reduce development speed. Use a compact, relational MVP schema first.
Unfinished UI Weakens employer perception even if backend work is decent. Prioritize a small number of polished pages.
Poor demoability A good project that is hard to explain loses value in interviews. Build around one clear end-to-end workflow with seeded data.
Partial writes Participant enrollment and visit generation can leave inconsistent state. Use a single Prisma transaction for multi-step writes.
Feature creep disguised as “senior improvements” Good ideas can still destroy schedule if added too early. Add only high-signal, low-cost improvements before launch.

10. Definition of Done

  • Core workflow works end to end
  • Demo can be completed in under 3 minutes without dead ends
  • Seed data exists for reliable walkthroughs
  • Study, site, participant, visit, document, and task flows are functional
  • Visit generation runs inside a single transaction
  • Status enums are used for stateful entities
  • Dates are stored in UTC and displayed correctly in the UI
  • Zod validation protects API boundaries
  • UI is clean, navigable, and coherent
  • Architecture and tradeoffs can be explained clearly in an interview
Bottom line: this project should present as a practical full-stack clinical research operations tracker that demonstrates strong data modeling, backend structure, workflow integrity, and product judgment using modern web technologies.
