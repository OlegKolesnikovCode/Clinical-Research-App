Executive Summary

A full-stack application modeling clinical research workflows with relational data modeling, controlled state transitions, transactional integrity, and validated API boundaries. This is a bounded, correctness-first system designed to enforce valid states, prevent partial system writes, model real-world workflows, and demonstrate disciplined engineering decisions.

Positioning: This is not a feature-heavy system. It is a constrained system that demonstrates correctness under failure, control over state transitions, and alignment with real operational workflows.

Problem Statement & Context

Clinical study operations often rely on spreadsheets, manual scheduling, and loosely enforced workflows. This creates inconsistent participant tracking, missed protocol events, and poor operational visibility.

Current Issues

  • Spreadsheet-based tracking
  • Manual scheduling
  • Weak workflow enforcement

Resulting Risks

  • Inconsistent participant tracking
  • Missed protocol events
  • Poor operational visibility

Solution Direction

  • Relational consistency
  • Workflow state rules
  • Deterministic execution paths

Scope Definition

In Scope (MVP)

  • Studies, Sites, Participants
  • VisitTemplates and ParticipantVisits
  • Documents (metadata only)
  • Tasks
  • Dashboard
  • Seed data
  • Validation with Zod
  • Status enums

Out of Scope

  • IRB workflows
  • eCRF systems
  • Electronic signatures
  • Billing systems
  • Full RBAC

Role Handling (Explicit Boundary)

Basic roles (Admin/User) are included. The UI demonstrates conditional rendering such as admin-only actions, while backend authorization remains intentionally simplified. This demonstrates authorization awareness without claiming production-grade security enforcement.

Goals, Functional Requirements, and Non-Functional Requirements

State Transition Rules (Central Integrity Layer)

Allowed

  • scheduled → completed
  • scheduled → missed
  • scheduled → cancelled

Disallowed

  • completed → scheduled
  • missed → completed
Enforcement mechanism: All status updates pass through API route handlers. Centralized validation logic enforces allowed transitions, and enum-backed states prevent invalid values.

Data Integrity

  • Foreign key relationships
  • Enum constraints
  • API validation with Zod
  • Transactional writes

Reliability

  • Multi-step operations use database transactions
  • Partial writes are prevented

Performance & Scalability

  • Responsive local usage
  • Avoid unnecessary full table scans
  • Index foreign keys and status fields
  • Pagination and indexing are the next steps

Data Model / Domain Model

User

Basic roles and UI-level action gating.

Study

Top-level protocol container.

Site

Operational study location.

Participant

Study enrollment and execution anchor.

VisitTemplate

Expected schedule definition.

ParticipantVisit

Actual execution state.

Document

Metadata-only file reference.

Task

Workflow tracking unit.

Key Modeling Decision

Template vs Instance Separation: VisitTemplate stores the expected schedule, while ParticipantVisit stores actual execution. This separates plan from reality, enables scalable scheduling logic, and prevents duplication.

Data Integrity Rules

  • Foreign keys enforce relationships
  • Enum fields enforce valid states
  • Participant uniqueness enforced at study level
  • Examples: screening number, subject identifier where applicable

Auditability Foundation

  • All entities include createdAt
  • All entities include updatedAt
  • createdBy is optional if added without complexity risk
