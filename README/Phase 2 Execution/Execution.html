Overview

This execution plan is structured around one constraint: maximize interview signal per unit effort. The build order prioritizes schema credibility, realistic data, validated APIs, atomic workflow execution, and demo reliability before optional polish.

Optimization rule: if a feature does not improve workflow integrity, data correctness, demo clarity, or interviewer confidence, delay it.

Architecture Flow

UI (Next.js) ↓ API Routes ↓ Zod Validation ↓ Prisma ORM ↓ PostgreSQL

Phase 1 — Foundation (Days 1–2)

GoalEstablish a stable backend foundation with a correct schema, working database connection, and clean migrations.
DeliverableRunning app, migrated database, usable Prisma Client, and base layout shell.
PriorityCorrectness first. Keep the schema minimal and explicit.
ValidationManual sanity checks in Prisma Studio and no migration errors.

Tasks

  • Create the Next.js app
  • Initialize git
  • Install Prisma, @prisma/client, Zod, and date-fns
  • Configure PostgreSQL using DATABASE_URL
  • Initialize Prisma
  • Write the full schema.prisma
  • Define models, relationships, and enums
  • Add onDelete: Cascade where appropriate
  • Add unique constraints for participant identity
  • Run migrations and generate Prisma Client
  • Create a base layout and simple navigation shell

Checklist

  • Next.js app running
  • Prisma configured
  • PostgreSQL connected
  • Schema complete
  • Enums implemented
  • Cascade rules added
  • Unique constraints added
  • Migration successful
  • Prisma Client generated
  • Base layout exists

Pitfalls

  • Overthinking schema design instead of keeping it minimal
  • Skipping enums and creating free-form state values
  • Forgetting cascade rules and breaking later reseeds

Validation

  • Run npx prisma studio
  • Manually create a Study and Site
  • Confirm relationships work without DB errors

Phase 2 — Seed Data (Day 3)

GoalCreate realistic data early so the UI can be built against real states instead of empty placeholders.
DeliverableA populated database that supports immediate frontend development and a stable demo path.
PriorityMake the app visually alive as early as possible.
ValidationSeed can run repeatedly without breaking relationships.

Tasks

  • Create prisma/seed.ts
  • Seed roles and 1–3 users
  • Seed 2–3 studies and sites
  • Seed visit templates
  • Seed participants with different statuses
  • Seed participant visits, tasks, and documents
  • Create records in FK-safe order
  • Run the seed script and verify data

Checklist

  • Seed script created
  • Seed script runs without errors
  • Studies exist
  • Sites exist
  • Participants exist
  • Visit templates exist
  • Participant visits exist
  • Tasks exist
  • Documents exist

Pitfalls

  • Wrong record creation order causing FK errors
  • Incomplete seed data creating a weak demo
  • Seed script not being rerunnable

Validation

  • Run seed multiple times without breaking the database
  • Inspect data in Prisma Studio
  • Confirm relationship chains are populated correctly

Phase 3 — API + Validation Layer (Days 4–5)

GoalBuild validated CRUD workflows for studies, sites, and participants.
DeliverableWorking backend endpoints plus minimal connected UI forms and list pages.
PriorityProtect the database at the API boundary using Zod.
ValidationInvalid requests fail safely before reaching Prisma.

Tasks

  • Create /api/studies, /api/sites, and /api/participants
  • Implement GET and POST routes
  • Add useful filtering by study or site where needed
  • Define Zod schemas for Study, Site, and Participant creation
  • Build Studies list, Create Study, Study Detail, Add Site, Participant list, and Enroll Participant UI
  • Handle validation and API errors clearly in the UI

Checklist

  • Study routes working
  • Site routes working
  • Participant routes working
  • Zod validation implemented
  • Validation errors handled
  • UI forms connected to API
  • Participant status enum used

Pitfalls

  • Skipping validation and letting bad data hit the DB
  • Overbuilding UI before the routes are stable
  • Not handling API failures

Validation

  • Send an invalid request and confirm it fails safely
  • Send a valid request and confirm the DB record is correct
  • Verify UI reflects newly created records

Phase 4 — Transactional Workflow Engine (Day 6)

GoalImplement atomic participant enrollment with automatic visit generation.
DeliverableA transactional enrollment flow that never leaves ghost records behind.
PriorityThis is the strongest engineering signal in the entire project.
ValidationForced failures do not create partial enrollment state.

Tasks

  • Fetch VisitTemplates for the selected study
  • Generate ParticipantVisits from template offsets
  • Use date-fns addDays for date math
  • Store generated dates in UTC
  • Wrap enrollment and visit creation in prisma.$transaction()
  • Prevent partial writes on any failure

Checklist

  • Visit templates fetched correctly
  • Visits generated correctly
  • Date logic works using date-fns
  • Transaction implemented
  • No partial writes on failure
  • UTC date logic used

Pitfalls

  • Manual date math bugs
  • Creating the participant outside the transaction
  • Timezone confusion between storage and display

Validation

  • Force a failure and confirm the participant is not created
  • Run a successful enrollment and verify visits are inserted
  • Check generated dates are correct and stored consistently

Phase 5 — Visit Execution Workflow (Day 7)

GoalTurn generated visits into an operational workflow with state changes.
DeliverableVisits can be viewed, completed, missed, or cancelled through the UI.
PriorityKeep the interface simple. A visit list is enough.
ValidationStatus transitions persist correctly and dates render cleanly.

Tasks

  • Build a participant visit list
  • Add visit status update route
  • Support scheduled, completed, missed, and cancelled states
  • Set actual completion date when needed
  • Format dates for display while preserving UTC storage

Checklist

  • Visit list renders
  • Status update route works
  • Complete visit action works
  • Status enums used
  • UTC dates display correctly

Pitfalls

  • Mixing local and UTC dates inconsistently
  • Overbuilding a calendar UI instead of a simple visit list
  • Using free-form status strings

Validation

  • Update a visit and verify DB state changes correctly
  • Confirm timestamps and statuses are accurate

Phase 6 — Documents & Tasks (Day 8)

GoalComplete the operational layer without introducing infrastructure drag.
DeliverableDocuments and tasks work end-to-end using simple metadata flows.
PriorityMock uploads only. Metadata matters more than storage.
ValidationCreated documents and tasks appear in the UI and update correctly.

Tasks

  • Create document metadata flow
  • Allow linking a document to a Study or Participant
  • Use a simple file URL or filename field for mock uploads
  • Create task flow: create, list, and update status

Checklist

  • Document creation works
  • Document list works
  • Task creation works
  • Task list works
  • Task update works

Pitfalls

  • Trying to integrate real file storage too early
  • Overcomplicating task logic

Validation

  • Create a document and verify it appears in the UI
  • Create and update a task successfully

Phase 7 — Dashboard & Polish (Day 9)

GoalMake the system presentable, coherent, and easy to demo.
DeliverableA dashboard, smooth navigation, and clean UX states.
PriorityPolish the demo path, not every edge of the UI.
ValidationDashboard metrics are correct and the app feels complete.

Tasks

  • Build dashboard metrics: total studies, active participants, upcoming visits, overdue tasks
  • Add loading states, empty states, and basic error states
  • Improve page spacing and readability
  • Verify seeded data supports a clean demo flow

Checklist

  • Dashboard metrics correct
  • UI readable
  • Empty states handled
  • Loading states handled
  • Navigation works smoothly

Pitfalls

  • Over-polishing cosmetic details
  • Adding extra features instead of improving the demo path

Validation

  • Dashboard reflects actual DB values
  • Main pages are easy to navigate and explain

Phase 8 — Hardening & Interview Prep (Day 10)

GoalMake the project reliable, explainable, and ready for interview pressure.
DeliverableA stable demo-ready system and a strong narrative for design decisions.
PriorityRepeat the demo path until it becomes boring.
ValidationThe project survives multiple complete walkthroughs without failure.

Tasks

  • Run the full demo path five times
  • Fix bugs and rough edges
  • Verify transactions, date handling, and validation
  • Clean up UI labels and wording
  • Finalize documentation
  • Prepare concise answers for relational design, transactions, UTC dates, enums, concurrency, and protocol amendments

Checklist

  • Demo path runs cleanly
  • No major bugs
  • Transaction logic verified
  • UTC handling verified
  • Validation verified
  • Documentation complete
  • Talking points ready

Pitfalls

  • Skipping repeated testing
  • Weak explanations of design tradeoffs
  • A demo that works once but not consistently

Validation

  • Run the workflow end-to-end repeatedly without hesitation
  • Answer pressure-test questions clearly and calmly

Final Rules

Daily Stop Rule

At the end of each day, ask: did this create visible progress, improve the demo, reduce risk, or raise interviewer confidence?

Daily Quality Check

Commit code, keep types clean, handle API errors, enforce validation, maintain UTC consistency, and avoid unnecessary tooling.

Final Demo Path

Dashboard → Study → Site → Participant → Visit generation → Visit completion → Document → Task → Dashboard recap.

Final filter: if a feature does not improve workflow integrity, data correctness, demo clarity, or interviewer confidence, do not build it yet.
